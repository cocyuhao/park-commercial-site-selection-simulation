# 公园商业选址仿真项目总计划

## 项目目标

建立一套可追溯、可复跑、可交接的公园商业经营决策系统，用于回答：

- 哪些业态存在供需缺口？
- 缺口发生在哪些空间节点和时间场景？
- 商家应该开在哪里、开几家、开多大、采用什么经营形式？
- 公园端预计收益、投资回收期、风险和敏感性如何？
- 最终如何通过一个面向行业专家的网站，让模型、证据、地图、场景和结论可交互使用？

## 2026-05-25 计划调整：Flowus 学习与专家网站化交付

### 本轮边界

- 本次仍只修改计划和交接文档，不落实代码、不新建网站、不生成 Postman collection、不运行新的 DeepSeek 任务。
- 已将本轮学习提炼为 `00_control/p6_expert_website_design_brief.md`，供未来 P6 调用。
- 用户提供的三个 Flowus 页面用于学习产品质感、AI 协作和网站呈现方法，不能替代本项目的证据链、模型和行业专家需求。
- 最终网站的用户是行业专家，因此第一屏应是可工作的决策驾驶舱，而不是营销落地页或泛泛介绍页。

### Flowus 学习结论

- 不让 AI 包揽全部决定：AI/DeepSeek/Codex 负责生成、归纳、草拟和加速，人负责定义“灵魂”、专家工作流、证据门禁和最终判断。
- 视觉参考可以来自 Mobbin、60fps、Spotted in Prod、Pinterest 等站点，但只能借鉴信息架构、动效节奏和质感，不直接照搬。
- “去 AI 味”的关键不是堆效果，而是让界面服务真实任务：地图、证据、参数、场景、收益、风险必须在同一个专家工作流中闭环。
- 图片、视频和微动效可以使用，但本项目应优先使用真实地图、流线、热力图、场景变化和仿真过程作为视觉资产；不做无关装饰。
- 后续可采用“灵感参考 -> 人定义专家工作流 -> Figma/原型 -> AI 辅助实现 -> 浏览器验证 -> 专家验收”的节奏。

### 为什么放在 P6

1. P1 仍在证据和供给核验阶段，不能因为网站外观提前包装不稳定结论。
2. P2/P3/P4 会逐步产出 API 契约、概率模型、真实公园参数和仿真结果，网站需要等这些接口和数据口径稳定后再做。
3. P5 先形成报告、PPT 和结论口径，P6 再把这些内容网站化，避免把网站当成替代分析的外壳。

### 专家网站信息架构草案

- 决策驾驶舱：展示候选业态、点位排序、收益区间、风险等级和证据完整度。
- GIS 地图页：展示公园边界、入口/节点、POI 供给、步行圈层、竞品、候选点和热力层。
- 场景实验室：切换保守/基准/乐观、天气、节假日、活动日、施工封路、竞品开业等场景。
- 人群仿真页：展示 persona、需求触发、选择概率、放弃率、外溢率和 Monte Carlo 不确定性。
- 证据追溯页：按推荐点、业态、指标反查 `evidence_ledger.csv`、POI 核验表、假设表和模型参数。
- 财务与风险页：展示租金、分成、改造投入、回收期、敏感性和风险折减。
- 专家审阅页：记录专家批注、待核验问题、人工调整理由和最终验收状态。
- 导出页：导出报告、PPT、CSV/XLSX、地图截图、Postman 验收附件和模型运行摘要。

### 初步技术规划

- 前端框架优先规划为 Next.js + React + TypeScript；是否落地等 P6 再确认。
- UI 组件优先考虑 shadcn/ui、Tailwind CSS 和可审计的自定义组件；避免把组件库默认样式直接当最终设计。
- GIS 展示优先考虑 MapLibre GL JS；复杂点层、热力层、流线和仿真轨迹可评估 deck.gl。
- 图表优先考虑 Apache ECharts；专家表格和证据清单可评估 TanStack Table。
- API 契约继续按前一节计划放在 P2/P4，Postman 只负责联调和回归验收，不作为证据来源。
- 浏览器/Playwright 验证放在真正建站后执行：检查地图非空、图表渲染、表格筛选、移动端布局、专家工作流和导出链路。

### 设计约束

- 专家网站应信息密度高、可扫描、可复核，避免大面积营销 hero、空泛宣传文案和装饰性卡片堆叠。
- 第一屏必须直接给出项目对象、核心推荐、证据完整度和主要风险，而不是先讲产品故事。
- 所有推荐卡片必须能点击追溯到证据、参数、场景和计算口径。
- 动效只用于表达状态变化、路径流动、场景切换和不确定性，不为“高级感”而单独存在。
- 任何无证据链支撑的文案都只能出现在假设池或待审阅区，不能进入专家结论区。
