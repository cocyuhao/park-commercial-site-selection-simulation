## 1. 本轮用户纠偏

- AI 工作台不能出现两个输入框；应使用一个统一输入框，同时兼容文字、图片、图纸、表格和方案文件。
- 不要把“提示词按钮”“位置说明框”等内部思路暴露给专家用户。
- 专家用户不是英文技术专家，页面主流程必须中文化、任务化。
- 交互必须从真实工作流出发：上传资料 -> AI 解析 -> 生成待复核候选 -> 人工确认 -> 进入节点/地图/资料闭合。
- 地图应接近高德/百度/苹果地图的基本使用体验：拖拽、缩放、图层、点位点击，而不是静态示意图。
- 高德 Key 不能进入前端；真实接入必须能被接口状态和浏览器截图验证。

## 3. 成熟 AI 工作台可借鉴模式

### 应复制的模式

- 单一 Composer：用户只面对一个输入区域；文字、上传、发送、加载状态都在这里完成。
- 附件随消息走：上传文件以附件 chip 展示在输入框上方或内部，发送时和本轮消息一起进入上下文。
- 输入框自动增高：短问题像单行输入，长意见自动展开，但不把页面撑坏。
- Enter 发送、Shift+Enter 换行。
- 思考中状态在消息流里展示，而不是全页面转圈。
- 系统边界用小状态说明，不把 prompt、字段名、内部模式按钮暴露给专家。
- 历史对话主导页面，左侧上下文只做轻量提示或可折叠抽屉。

### 当前 P6 应删除或弱化的交互

- 删除独立“位置说明”输入框。
- 删除“解释草案边界 / 列补数清单 / 转成模型修改项”这类显式提示词按钮。
- 不让用户选择“保存专家意见”还是“发送给 AI”；发送时自动登记为待复核专家输入。
- 不把 `needs_review`、`pending_conversion`、`blocked_until_source_received` 大面积放在专家主视图。

## 4. 专家真实工作流

### AI 工作台

1. 专家打开 AI 工作台。
2. 页面显示当前节点和边界，但不占据主要空间。
3. 专家在一个框里输入意见，或点 `+` 上传图纸/图片/数据表。
4. 系统把本轮输入保存为待复核专家输入。
5. 后端调用 DeepSeek，输出：
   - 这条意见影响哪些节点/假设；
   - 能闭合哪个 P3 资料缺口；
   - 仍缺什么资料；
   - 下一步问合作方什么；
   - 明确 `待复核 / 非最终`。
6. 人工确认后，才进入节点、地图或资料闭合状态。

### 上传资料

1. 上传文件进入资料池，不直接变成结论。
2. AI 先识别文件类型和可能用途。
3. 生成节点/点位/缺口候选。
4. 人工确认候选是否有效。
5. 只有确认后的内容才写入可用输入池。

### P3 资料闭合

- 页面不应只告诉用户“被阻塞”，而应给出：
  - 需要什么资料；
  - 可以从哪里上传；
  - 上传后谁复核；
  - AI 能先做什么；
  - 还不能做什么最终判断。

## 5. 地图技术判断

### 当前事实

- 用户提供的新高德 Web Service Key 已写入本地 `.env`。
- 后端 `/api/amap/static-map` 已能返回 `image/png`。
- 这只证明静态图服务可用，不代表 JS 交互地图已完成。

### 安全边界

- Web Service Key 不应直接放进前端。
- 若要使用高德 JS API，需要按官方方案准备浏览器端 JS Key 和安全密钥代理/安全配置。
- 如果只使用 Web Service Key，则应继续由后端代理搜索、静态图、POI 查询等能力。

### 可选路线

1. 后端 Web Service + 前端自定义地图交互：安全、可控，但交互体验很难完全像高德。
2. 高德 JS API：体验最接近高德，但需要正确的 JS Key、安全密钥和域名配置。
3. MapLibre/deck.gl：适合正式系统和仿真图层，但不是“1:1 高德”。

当前建议：短期用 Web Service 验证静态底图与 POI 查询；下一阶段单独申请/配置高德 JS API 浏览器端能力，做真正拖拽缩放地图。
