# P6 专家网站设计简报

> 本文件用于未来进入 P6“专家网站化交付”时调用。它只沉淀设计建议、产品判断和工具路线，不代表当前已经开始实现网站。

## 1. 当前边界

- 当前项目仍在 P1，尚未进入 P2/P3/P4/P5/P6。
- 本文件来自 2026-05-25 至 2026-05-26 的规划讨论，目标是把 Flowus 学习、外部工具资料和专家网站建议沉淀成可复用文档。
- 本文件不要求现在新建网站、不改前端代码、不创建 Postman collection、不运行新的 DeepSeek 任务。
- P6 只有在证据链、模型输入、仿真输出、报告结论和 API 契约稳定后才进入。

## 2. P6 的真实目标

P6 不是做一个“好看官网”，而是把 P1-P5 的成果变成行业专家能实际使用的决策系统：

- 能看懂：一眼看到推荐业态、推荐点位、收益区间、风险等级和证据完整度。
- 能复核：每个结论都能追溯到 `evidence_ledger.csv`、POI 核验表、参数表或明确假设。
- 能调参：专家可以调整人群、场景、转化率、租金、分成、天气、节假日、活动等参数。
- 能比较：支持 baseline、候选方案、保守/基准/乐观、压力场景之间对比。
- 能导出：导出报告、PPT、CSV/XLSX、地图截图、Postman 验收附件和模型运行摘要。

## 3. Flowus 学习提炼

用户提供的三个 Flowus 页面可以提炼为一个设计方法，而不是一组要照搬的页面。

### 3.1 方法核心

- 不让 AI 包揽全部决定。AI 可以生成、归纳、草拟、加速，但“产品灵魂”、专家工作流、证据门禁和最终判断必须由人定义。
- 先定工作流，再定视觉。先明确专家如何看结论、查证据、改参数、看风险，再选择页面布局、组件和动效。
- 用真实素材消除“AI 味”。本项目的真实素材不是抽象背景，而是地图、边界、入口、步行圈层、POI、热力、仿真轨迹、证据表和财务曲线。
- 动效只服务理解。路径流动、场景切换、参数变化、概率分布变化可以动；纯装饰动效不进入专家系统。
- 参考网站只能借鉴质感和结构，不直接复制。Mobbin、60fps、Spotted in Prod、Pinterest 等适合做灵感库，但要重写为公园商业选址专家场景。

### 3.2 对本项目的转化

Flowus 提到的“功能生成 -> 人定义灵魂 -> 素材提取 -> 原型组装 -> AI 代码”在本项目中转化为：

1. 功能生成：由 P2-P4 产生模型 API、仿真输出、地图数据和证据查询能力。
2. 人定义灵魂：确定行业专家最关心的是“为什么这里、怎么算、风险在哪、证据够不够”。
3. 素材提取：使用真实地图、POI、路径、热力、表格、报告结论和模型输出作为视觉资产。
4. 原型组装：先做信息架构和关键页面原型，确认专家工作流闭环。
5. AI 辅助实现：Codex/Copilot/Claude 等只在需求清楚后辅助实现页面和交互。
6. 浏览器验证：真正建站后必须用浏览器检查地图、图表、表格、布局、导出和移动端。

## 4. 三种路线比较

| 路线 | 做法 | 优点 | 风险 | 结论 |
|---|---|---|---|---|
| A. 报告门户 | 把报告、PPT、地图截图和表格做成可浏览网站 | 成本低，适合展示 | 交互弱，专家不能调参，像资料库 | 可作为 P6 的最低保底，不作为主路线 |
| B. 专家决策驾驶舱 | 第一屏展示推荐、证据、地图、风险和场景入口 | 最符合行业专家使用方式，可复核、可比较 | 需要稳定数据接口和页面设计 | 选为主路线 |
| C. 仿真实验室优先 | 先做参数调节和 Agent 动态仿真界面 | 技术感强，适合演示复杂模型 | 容易过早包装不成熟模型，忽略证据链 | 放在 P6 中后段，不能替代驾驶舱 |

最终选择：以 B“专家决策驾驶舱”为主，吸收 A 的导出与归档能力，等 P4 仿真稳定后再加入 C 的实验室能力。

## 5. P6 进入条件

进入 P6 前至少需要：

- `evidence_ledger.csv` 已形成可用于报告的 checked 指标集合。
- 供给侧 POI、入口/节点、边界、运营授权和经营字段已有明确状态。
- P2 的缺口计算器、人群概率原型和 API 契约已稳定。
- P3 的真实目标公园数据已接入并记录缺口。
- P4 的 GIS/游客 Agent 仿真已有 baseline 和候选方案输出。
- P5 的推荐结论、财务测算、风险解释和报告口径已被复核。
- Postman collection/report 如存在，只作为 API 验收附件，不作为证据来源。

## 6. 专家用户与核心任务

目标用户是行业专家、规划/招商/运营/投资评审人员，不是普通游客。

他们进入网站时通常要完成这些任务：

- 判断哪些业态值得做，优先级如何。
- 判断推荐点位是否真的有需求、有供给缺口、有可达性、有经营可行性。
- 检查收益测算是否过于乐观。
- 调整关键参数，看方案是否仍成立。
- 找出需要现场核验或官方确认的问题。
- 把结论导出给汇报、评审或内部决策使用。

## 7. 信息架构

### 7.1 决策驾驶舱

第一屏必须展示：

- 项目名称、目标公园、当前数据版本和模型版本。
- 推荐业态/点位排序。
- 收益区间、风险等级、投资回收期。
- 证据完整度、待核验数量、关键假设数量。
- 当前场景：保守/基准/乐观/压力场景。
- 进入地图、场景、证据、财务和导出的入口。

不允许第一屏只放大标题、营销文案或抽象背景。

### 7.2 GIS 地图

地图页应支持：

- 公园边界、入口/节点、停车/交通节点。
- 园内候选供给、园外竞品供给、POI 分类图层。
- 步行 3/5/10 分钟圈层或路径。
- 热力、流线、仿真轨迹或需求外溢方向。
- 点击点位后展开证据、经营字段、路径可达和风险状态。

地图数据必须标注来源、坐标系和边界限制。OSM 边界不能被说成官方规划红线。

### 7.3 场景实验室

场景实验室应支持：

- 保守、基准、乐观三档。
- 天气、节假日、活动日、施工封路、竞品开业、临时摊位等压力场景。
- 用户临时提出的个性化需求进入“假设池”，不能直接变成事实参数。
- 每次场景运行保存参数版本、随机种子、样本数和输出摘要。

### 7.4 人群仿真

人群仿真页应展示：

- `persona` 分群：亲子、运动、白领、老人、游客、夜间消费人群等。
- 需求触发：时间、天气、同行人数、疲劳、排队、儿童需求、突发购买。
- 概率输出：业态选择概率、店铺选择概率、转化率、放弃率、外溢率、停留时间。
- 不确定性：均值、分位数、置信区间或 Monte Carlo 分布。

### 7.5 证据追溯

证据页应支持：

- 按推荐点、业态、指标、来源文件、证据等级筛选。
- 每个结论能反查证据 ID、原始来源、页码/接口、抽取方法、校验状态。
- 区分强证据、弱证据、模型估算和人工假设。
- DeepSeek 输出只显示为 `draft` 或 `needs_review`，不得进入 checked 结论。

### 7.6 财务与风险

财务页应支持：

- 客流捕获、转化率、客单价、租金、分成、改造投入、运营成本。
- 保守/基准/乐观收益区间。
- 回收期、敏感性分析和风险折减。
- 风险来源：数据缺口、授权不确定、季节波动、天气、竞品、审批、施工。

### 7.7 专家审阅

专家审阅页应支持：

- 专家批注。
- 人工调整参数的理由。
- 待核验清单。
- 结论是否可进入报告/汇报/招商建议。

### 7.8 导出

导出页应支持：

- 报告、PPT、CSV/XLSX、地图截图。
- 当前场景参数和模型运行摘要。
- API 验收附件和 Postman report。
- 导出文件必须带数据版本、模型版本和生成时间。

## 8. 视觉与交互原则

- 风格方向：专业、克制、可信、信息密度高，接近“规划评审终端 + 投研驾驶舱 + 运营指挥台”。
- 页面结构：侧边导航 + 顶部项目状态 + 主工作区 + 右侧证据/参数抽屉。
- 色彩：不要做单一蓝灰、紫色渐变或营销感大色块；用中性底色承载地图和表格，用少量高对比色表达风险、推荐和状态。
- 字体：不追求花哨，优先保证中文可读、数字清晰、表格密集场景不挤压。
- 组件：按钮、图层开关、筛选器、滑杆、分段控件、风险标签、证据抽屉、参数面板必须稳定，不因内容变化导致布局跳动。
- 动效：只用于路径流动、图层切换、参数变化、场景对比和加载状态；避免无意义大动画。
- 移动端：不追求完整专业操作，只保证查看核心推荐、风险、证据摘要和导出状态。

## 9. 工具路线

这些工具只作为 P6 候选，真正落地前必须重新查官方文档和版本：

- Next.js + React + TypeScript：适合做专家网站、路由、数据界面和 API 集成。
- shadcn/ui：适合快速构建可控的 React 组件，但不能直接套默认样式。
- Tailwind CSS：适合建立设计 token、布局和响应式样式。
- MapLibre GL JS：适合开放地图、边界、点位和交互图层。
- deck.gl：适合大规模地理图层、热力、流线、轨迹和仿真可视化。
- Apache ECharts：适合收益、风险、敏感性、概率分布和对比图。
- TanStack Table：适合证据清单、模型输入表、复核队列和专家筛选。
- Postman CLI：适合 API 契约、smoke test 和回归验收，不保存真实 Key。
- Browser/Playwright：适合 P6 实现后的地图非空、图表渲染、布局和交互验证。

## 10. AI 使用边界

- Codex/Claude/Copilot：可用于页面实现、组件重构、浏览器验证和缺陷修复。
- DeepSeek：可用于整理个性化需求、生成场景草稿、解释模型输出、提示风险点。
- 本地 Python：仍负责所有计算、仿真、排序、收益、敏感性和证据门禁。
- AI 不得直接给最终推荐、最终收益结论、checked 证据入账或凭据处理。

## 11. P6 验收清单

进入 P6 完成状态前，至少满足：

- 第一屏是决策驾驶舱，不是营销页。
- 每个推荐点都能追溯证据、参数、场景和计算口径。
- 地图、图表、表格在浏览器中真实渲染，不是静态占位。
- 场景切换能保存参数版本和输出摘要。
- 专家批注和人工调整能记录理由。
- 导出内容带数据版本、模型版本和生成时间。
- Postman/report 只作为验收附件，不含真实 Key。
- 没有证据链支撑的内容只出现在假设池或待审阅区。
- 移动端不重叠、不遮挡、不出现无法阅读的核心信息。
