# 老板六份方法资料与外部论文的统一方向矩阵（2026-06-04）

> 状态：方法整合稿，供后续实现和交接使用。  
> 边界：本文不宣称 P4 完整仿真已经完成，不把任何 DeepSeek 输出升级为 checked 证据。

## 0. 先回答当前纠偏

用户的最新纠正是关键约束：DeepSeek 便宜，但不能假设它足够聪明；六份老板资料不是零散参考，也不能武断地说它们天然合成了一个完整系统。更准确的判断是：这些资料的方向是一致的，都在约束我们不要让 LLM 直接编故事、直接打分或直接替代真实校准；它们共同提示我们，真实资料、人的状态、空间行为、商业选择、仿真验证和业务报告必须被放进一条可解释、可复核的工作链。

因此本项目后续不能再走“读一点材料就马上补代码”的节奏。正确顺序应是：

1. 先把方法资料和高质量论文转成同一方向下的设计约束。
2. 再把每个层级转成数据 schema、可复跑脚本、校验门禁和页面呈现。
3. 最后才让 DeepSeek 在受限位置做便宜的批量语义工作。

刚才新增的 persona/behavior CSV 和 `60_model/simulation/persona_behavior.py` 只能视为草稿候选，不能视为最终设计完成。

## 1. 三条路线比较

| 路线 | 优点 | 风险 | 判断 |
|---|---|---|---|
| LLM-first，让 DeepSeek 直接模拟游客、打分、写结论 | 成本低，开发快，页面容易看起来“智能” | 黑箱、不可复跑、容易把 plausible 文本当事实；会制造神秘分数和伪仿真 | 拒绝作为主线，只能做草稿层 |
| 传统 GIS/POI/TGI 公式优先 | 可解释、便于门禁、容易落地 | 解释不了人的状态、绕行、疲劳、同行、排队、放弃和冲动消费 | 作为底座，但不够完整 |
| 证据层 + 人群状态 + 行为程序 + 本地仿真 + DeepSeek 受限语义层 | 能把方向一致的方法约束落到老板资料、真实数据、用户体验和报告输出上 | 需要更慢、更严谨的 schema 和验证设计 | 采用 |

最终路线：**本地 Python 和证据链是主干；DeepSeek 是便宜语义工人；Codex/人工负责架构、门禁、复核和最终判断。**

## 2. 同一方向下的工作框架

### 2.1 证据层

输入来源：

- `40_quality_evidence/evidence_ledger.csv`
- PDF 原生表格、TGI、客流、热门到访 POI
- 高德 POI、路径、边界、截图或后端代理结果
- 用户上传的计划、DOCX、PDF、图片、现场反馈
- 老板六份方法资料
- PPT/报告材料的表达方式

规则：

- `checked` 才能作为强证据。
- `needs_review` 只能作为草案或待复核线索。
- PPT 内容不当强证据，只学习角度、语言、层级、商业表达。
- DeepSeek 不能写入 checked 证据。

### 2.2 人群状态层

来自 HumanLM 的核心启发：不要模仿表面回答，要对齐潜在状态。公园商业语境下，至少应保留：

- 到访目的：运动、亲子、社交、通勤、文化参观、康养、夜间活动。
- 身体状态：疲劳、口渴、饥饿、携带物、照护儿童或老人。
- 时间压力：赶路、等人、活动前后、可停留。
- 预算和价格敏感：快取、品质停留、平价补给、纪念消费。
- 同行关系：独行、情侣、亲子、老人陪同、团队。
- 绕行意愿：愿意多走多少米、是否需要遮阴/座位、是否怕排队。
- 风险容忍：天气、拥挤、营业不确定、路线不清晰。

页面影响：用户不应该只看到“亲子家庭/白领/游客”这种空标签，而应该能看到状态为何导致某种节点建议。

### 2.3 行为程序层

来自 ROTE 的核心启发：很多日常行为可写成可编辑的行为程序，而不是每次让 LLM 重新编故事。

每条行为程序至少包括：

- 适用人群
- 触发条件
- 初始状态
- 路径偏好
- 停留动作
- 消费或服务动作
- 放弃条件
- 外溢条件
- 概率参数
- 证据来源
- 待复核项

例子：

- 晨跑补给：早高峰、运动人群、跑后口渴、偏好近距离饮水/运动饮料，排队过长则放弃。
- 亲子休息：周末、儿童疲劳、需要遮阴/座位/轻餐，距离过远或无座位则外溢。
- 午间咖啡：工作日、周边办公人群、时间压力高、偏好快取，超过绕行阈值则放弃。

DeepSeek 可以生成这些脚本草稿，但脚本必须进入本地 schema 和校验，不得直接成为最终路线。

### 2.4 空间运动层

来自 Social Force、RVO、MATSim 的共同启发：空间仿真不是“地图上点会动”这么简单。真实可用的空间层至少需要：

- 网络或路径：入口、节点、道路、可达距离、障碍。
- 容量与拥挤：排队、座位、通道宽度、节点承载。
- 时间：工作日/周末/节假日/活动前后/天气。
- 行为活动链：不是从 A 到 B 一步，而是多目标、多停留、多放弃。

当前本地 `60_model/simulation/engine.py` 更接近结构化 dry-run，不是完整空间运动仿真。后续若要做人群仿真，不能只靠静态地图或 JS 动画，应由 Python/仿真模块先产生可复跑结果，再由前端展示。

### 2.5 消费选择与业态决策层

来自 Huff model、logit、retail location ABM 和 GameLook/SSR 的共同启发：

- 不要让 AI 直接打 1-100 分。
- 先生成自然语言动机、担忧、放弃理由。
- 再把这些理由映射到选择概率、优先级、证据缺口。
- 业态之间有互补和替代关系，不能单点孤立看节点。

节点页面应从“分数”改为：

- 推进优先级：优先推进 / 补资料后判断 / 仅做空间观察 / 暂缓。
- 为什么有人会来。
- 为什么有人会买。
- 为什么有人会放弃。
- 现在最具体的建议。
- 哪些证据会改变判断。
- 当前置信状态。

如果保留分数，只能作为折叠后的辅助解释，不应成为主视觉。

### 2.6 校准与验证层

来自社区高清社会仿真论文的核心启发：可信仿真必须有状态-行为-证据链一致性和宏观统计一致性双门禁。

本项目可迁移的验证指标：

- 微观：行为程序是否符合状态、路径、排队和服务逻辑。
- 宏观：客流峰值、时段曲线、节点停留、业态需求、POI 供给是否接近真实数据。
- 时间序列：SARIMA 或趋势一致性检查。
- 分布差异：KL-Divergence。
- 形态相似：SSIM，可用于热力图或时空栅格。
- 路径/曲线相似：DTW 或相关系数。
- 产品验证：Selenium 多轮人类路径、截图、人眼检查、控制台错误。

没有真实客流、DWG 几何、转化率、收益/成本、运营授权时，不得宣称完整仿真完成。

### 2.7 UI 与报告层

用户反复强调“人类视觉”和“客服端可用性”，这不是表面美化，而是模型可信度的一部分。页面应帮助业务人员理解：

- 现在能确认什么。
- 现在只是草案的是什么。
- 为什么系统这么判断。
- 还缺什么资料。
- 下一步能做什么动作。

报告应像商业工作报告，而不是 AI 日志；AI 工作台应像项目沟通与报告工作台，而不是测试面板。

## 3. DeepSeek 的受限位置

DeepSeek 可做：

- 批量资料摘要、主题分类、候选证据抽取。
- 潜在状态候选生成。
- 行为程序草稿生成。
- 节点建议的自然语言解释草稿。
- 报告语言初稿。
- LLM judge 型状态-行为一致性草评。

DeepSeek 不应做：

- 逐游客实时仿真。
- 最终节点排名。
- 最终收益预测或 ROI。
- checked 证据入账。
- P4 完整仿真完成声明。
- 无校准数据时的因果判断。

实现约束：

- 所有输出必须带 `output_status=draft/needs_review`。
- 请求必须有 schema、JSON 示例、字段白名单。
- 使用 JSON Output 时要处理空 content、截断、格式错误和重试。
- 低成本不等于低门禁；DeepSeek 的每个任务都要有本地验证脚本。
- 适合批处理和缓存，不适合在仿真循环中频繁调用。

## 4. 用户纠正转成项目规则

| 用户纠正 | 转成规则 |
|---|---|
| “DeepSeek 不是很智能” | DeepSeek 是语义工人，不是主模型；所有结果需 schema、门禁、复核 |
| “你没有把我发的东西往一件事上看” | 六份方法材料方向一致，必须放到同一条可解释、可复核的工作链里，但不能武断说天然合成一个完整系统 |
| “看的浅，没有联想思考” | 每篇材料要回答：输入、输出、验证、禁用场景、落地模块 |
| “分数意义不详，建议最重要” | 节点主视觉改为优先级、解释、建议、补证动作，分数折叠为辅助 |
| “不要空想” | 所有实现必须连到老板材料、论文、本地证据或可验证用户路径 |
| “资料处理和搜索不能断” | 研究产物必须落盘到 `10_research/`，交接文件记录当前状态 |

## 5. 已确认高价值外部资料

以下不是完整文献综述，而是当前筛出的可落地方法来源：

- Generative Agents: memory / reflection / planning，用于 AI agent 记忆与工作台交互模式。
- ROTE / Modeling Others' Minds as Code：用于行为程序层。
- HumanLM：用于潜在状态层。
- Real world community oriented high-definition social simulation：用于 RL+LLM、宏观校准和真实性门禁。
- Generative Agent Simulations of 1,000 People：用于长访谈记忆、合成用户偏差和早期市场反馈边界。
- LLM behavioral simulation causal inference critique：用于防止把 AI 输出当因果。
- Critical review of generative ABM：用于防止只看 believability，不做 operational validity。
- Social Force Model / RVO / MATSim：用于空间运动、拥挤、活动链。
- Huff model / logit / retail location ABM：用于消费选择、距离衰减、互补业态和优先级映射。

## 6. 后续实现顺序

1. 先修正文档口径：当前是方法原型和 dry-run，不是完整 P4 仿真。
2. 重构节点解释：主展示优先级、建议、证据缺口，隐藏裸分数。
3. 设计人群状态 schema 和行为程序 schema，先不接入完整仿真循环。
4. 设计 DeepSeek 任务边界：每个任务明确输入、输出、schema、验证脚本和失败处理。
5. 将资料池关联到状态/行为/空间/供给/校准，而不是单纯文件列表。
6. 再决定当前草稿 CSV 和 `persona_behavior.py` 是保留、改写还是删除。
