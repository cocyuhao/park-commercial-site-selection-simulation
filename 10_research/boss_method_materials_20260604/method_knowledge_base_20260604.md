# 老板方法资料知识库（2026-06-04）

本文件只处理本轮放入 `老板资料/` 的 6 份方法资料，以及为理解它们补充的外部论文和成熟产品线索。旧的 PDF、CSV、证据台账是仿真数据来源；PPT 只学习汇报角度、语言节奏和信息层级，不把 PPT 内容当作强证据。

## 一、先给结论：当前不能再把 dry-run 叫完整仿真

当前 `60_model/simulation/engine.py` 做的是结构化干跑：按 POI 类别、边界状态和门禁缺口分组，输出 `why_blocked`、`missing_required_fields`、`next_data_needed`。它有价值，但不是用户要求的“人物行为、路线、时间、消费、供需与概率仿真”。

因此后续文档和页面要改口径：

- 可以说：结构化干跑、供需缺口原型、待复核优先级。
- 不能说：完整 P4 人物仿真已完成、真实收益预测、最终排名。
- 原因：老板资料强调“状态对齐、行为程序、真实数据校准、宏观统计验证”，这些当前都还没有闭合。

## 二、六份老板资料的吸收结果

### BOSS-METHOD-001 长图 / GameLook 文章

这份材料不是让我们照搬游戏行业，而是给出一个判断框架：AI 模拟用户可以降低早期调研成本，但必须用真实数据和真实排名校准，不能把 AI 输出当真人反馈。

项目采用：

- 不直接让 AI 输出“1-100 分”或“好/不好”。先让虚拟用户生成自然语言：为什么经过、为什么停留、为什么购买或放弃，再由系统把自然语言映射为缺口概率、讨论优先级或待补资料。
- 建立“真实校准”要求：客流/TGI/POI/现场记录/运营反馈至少一个层面能回查，否则只保留为假设。
- 分数不能裸露为神秘数字。页面应优先显示“现在建议怎么做、为什么这么做、还缺什么证据”。

项目不采用：

- 不把文中“85% 准确率”写成本项目承诺。
- 不把合成消费者替代真人、现场核验和管理方确认。

### BOSS-METHOD-002 Modeling Others' Minds as Code / ROTE

核心启发：许多日常行为不是每次都深度推理，而是遵循脚本。例如“晨跑 5 公里 -> 找水/补给 -> 离开或拉伸”“亲子游玩 -> 孩子疲劳 -> 休息/轻餐”。ROTE 用 LLM 生成行为程序候选，再用概率推断处理不确定性。

项目采用：

- 新增“行为程序/路线脚本”概念，而不是只做人设。
- 每个脚本至少包含：适用人群、触发条件、路线偏好、停留动作、消费/服务需求、放弃条件、数据来源、校准状态。
- DeepSeek 可以生成脚本草稿，但 Python 负责参数、概率、门禁和复跑。

项目不采用：

- 当前不做完整程序合成引擎，不承诺贝叶斯推断已落地。
- 不让 LLM 单独决定最终路线和收益。

### BOSS-METHOD-003 HumanLM

核心启发：用户模拟不能只模仿表面回答，而要对齐潜在状态。论文里的状态维度包括 stance、emotion、communication、values 等，本项目要转换成公园商业语境。

项目采用的状态维度：

- 到访目的：运动、亲子、社交、通勤、文化参观、康养、夜间活动。
- 身体状态：疲劳、口渴、饥饿、携带物、儿童/老人照护压力。
- 时间压力：赶路、可停留、等人、活动前后。
- 预算与价格敏感：快取、品质停留、平价补给、纪念消费。
- 同行关系：独行、情侣、亲子、老人陪同、团队。
- 绕行意愿：愿意多走多少米、是否需要遮阴/座位、是否怕排队。
- 风险容忍：天气、拥挤、营业不确定、路线不清晰。

项目不采用：

- 不训练专用 HumanLM。
- 不把“年龄、收入、性别”三项当成完整画像。

### BOSS-METHOD-004 GameLook 转换 PDF

这份 PDF 是长图内容的文字版，重点再次强化两个方法：1000 人智能体需要丰富记忆素材，SSR 类方法要求从自然语言映射评分，而不是直接问分。

项目采用：

- 人物画像应绑定资料记忆：真实客流、TGI、POI、报告片段、用户上传计划、现场观察，而不是只有一句“白领/亲子”。
- 对话和报告里应该出现“待复核判断”，并说明依据来自哪些资料。
- 测试时不能只看接口通过，要模仿人类连续使用：上传资料、编辑状态、运行场景、追问、生成报告。

项目不采用：

- 不因为文中公司和游戏案例看起来成功，就忽略本项目的公园、空间、运营授权和门禁差异。

### BOSS-METHOD-005 中文摘要 DOCX

这份文件是 Cities 论文的中文速览。它把重点讲得很清楚：真实社区数据、3000 智能体、双重奖励、手机使用模式校准、SARIMA 验证。

项目采用：

- 把“状态-行为-证据链一致性 + 宏观一致性”作为仿真验证原则。
- 状态-行为-证据链一致性：单个仿真对象的状态、路线、时间、场景、消费/放弃条件是否能被资料、规则和可复跑计算共同解释。
- 宏观一致性：总体客流、时间峰值、业态需求、POI 供给和真实资料是否一致。

项目不采用：

- 不把“手机使用模式 90% 相关”迁移成本项目准确率。
- 不用中文摘要代替英文全文判断。

### BOSS-METHOD-006 Cities 2026 全文

核心启发：高质量社会模拟不是“让 AI 想象很多人”，而是要有真实环境、真实人口/行为先验、动作空间、奖励机制、训练/验证流程和宏观数据对齐。

项目采用：

- 分层校准：
  1. 人群结构校准：人群比例和 TGI 来自证据台账或用户上传资料。
  2. 时间分布校准：日峰值、时均峰值、节假日/工作日差异。
  3. 空间校准：入口、节点、路线、边界、POI 距离和可达性。
  4. 行为校准：停留、排队、购买、放弃、外溢。
  5. 宏观校准：仿真总量和真实客流/TGI/消费/现场反馈对比。
- 动作空间必须显式：走、停、买、排队、绕行、放弃、去竞品、返回主路、求助/休息。

项目不采用：

- 当前不做 Unreal Engine 高清社区复刻。
- 当前不声称 RL 训练已完成。

## 三、同事 POI/TGI 仓库的吸收边界

同事的 `POI_TGI_Calculator` 更像“供需缺口计算器”，不是完整人物仿真。可吸收部分：

- `demand_score - supply_score` 与标准化 `gap_index` 的解释结构。
- 消费业态与服务设施分开处理。
- 建议携带 `source / confidence / evidence / review_status`。
- PDF 指标、游客画像、AMap POI、缺口优先级之间的编排思路。

必须改造的部分：

- `TouristAgent` 不能只凭年龄、收入、目的生成 TGI；需要状态维度、行为程序和真实数据约束。
- `gap_index` 不能裸露给用户当最终评分；应映射成“当前推进优先级 / 待补资料与决策动作”。
- 建议必须具体：放在哪里、服务什么人、什么时段、什么约束、下一步核验什么。

不吸收的部分：

- 不整仓覆盖本地。
- 不复制其环境变量和 API Key 命名口径。
- 不把同事的算法输出升格为 checked 结论。

## 四、外部文献如何进入本项目

本轮已生成 `external_literature_review_20260604.csv`，包含 21 条相关论文/方法线索。它们的作用分层如下：

- LLM 用户仿真：HumanLM、Out of One Many、Using LLMs to Simulate Multiple Humans、1000 People。
- 生成式社会智能体：Generative Agents、LLM agent survey。
- 方法风险：因果推断批判、ABM 批判综述、身份扁平化、思维多样性下降。
- 行人/路径仿真：Social Force、RVO、MATSim。
- 商业/空间选择：Huff、Gravity、Logit、零售互补品 ABM。

这些文献给本项目的共同结论：

- LLM 适合生成状态解释、行为候选、报告草稿和假设整理。
- 数值、概率、收益、排序和校准必须由可复跑模型和证据门禁控制。
- “多智能体”不等于真实多样性，必须用真实人群结构和状态参数制造差异。
- 商业选址不能只看 POI 数量；要看时间、路线、需求触发、距离衰减、竞品、互补、运营可行性。

## 五、要落地的系统对象

后续系统至少应有这些可增删改查对象：

- `persona_group`：人群组，不是单个固定人物。
- `latent_state_profile`：每类人群在不同场景下的状态。
- `behavior_program`：路线/动作/消费脚本。
- `demand_trigger`：需求触发条件。
- `supply_opportunity`：供给机会，不等同于最终推荐。
- `scenario`：天气、时间、节假日、活动、施工、竞品变化。
- `validation_target`：要对齐的真实数据指标。
- `assumption_gap`：缺口和假设池。

这些对象都要给用户编辑权。DeepSeek 可以帮助生成草稿，但用户和证据门禁决定是否采用。

## 六、对页面和报告的直接要求

- 项目总览不应显示死状态；要显示位置、资料、节点、POI、AI理解、报告可生成、缺口是否闭合。
- 节点卡片不应突出神秘分数；应突出推进优先级、依据、建议、待补资料。
- 资料池要像产品功能：采用、放弃、删除、解析状态、引用位置。
- AI 工作台不能替用户做最终判断；它应该解释“当前能判断什么、不能判断什么、下一步怎么补”。
- 报告语言应像业务汇报：摘要、关键依据、当前缺口、待复核判断、推进事项、附录。

## 七、当前必须记录的风险

- P4 完成口径历史上存在矛盾，需要在 `progress.md` 和后续报告中修正。
- 真实 DWG 几何、入口/节点、客流分时、转化率、收益成本、运营授权仍未闭合。
- 高德 POI 只能说明空间供给线索，不能自动证明园内合法经营或可落位。
- DeepSeek 输出必须保持 `draft / needs_review`，不能进入 checked 证据。
- UI 测试通过不等于模型真实，必须增加“真实数据校准测试”。
