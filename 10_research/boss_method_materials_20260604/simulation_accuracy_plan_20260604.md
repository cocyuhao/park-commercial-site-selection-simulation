# 人物仿真准确性方案（基于老板资料与本地项目，2026-06-04）

目标不是做一个看起来会跑的动画，而是建立能被业务人员理解、能被同事复跑、能被证据纠错的人物仿真框架。

## 一、准确性的定义

本项目里的“准确”分四层：

1. 状态准确：虚拟人群的目的、时间压力、疲劳、预算、同行关系和绕行意愿与真实资料一致。
2. 行为准确：路线、停留、消费、排队、放弃和外溢行为符合状态和场景。
3. 宏观准确：总客流、时段峰值、业态需求、POI供给和真实数据趋势一致。
4. 决策准确：输出能指出下一步该补什么资料、做什么运营动作，而不是只给分数。

## 二、模型分层

### 1. 证据层

输入：

- `40_quality_evidence/evidence_ledger.csv`
- 用户网页上传资料
- 高德 POI / 路径 / 边界
- 现场或官方核验结果
- 假设池

规则：

- 真实数据、模型估算、人工假设分开。
- `checked` 才能做强证据；`needs_review` 只能做草案。
- PPT 只学语言和角度，不作为数据来源。

### 2. 人群状态层

每个 `persona_group` 至少包含：

- 群体名称
- 适用场景
- 证据来源
- 到访目的
- 时间压力
- 疲劳/饥渴/照护压力
- 同行结构
- 预算与价格敏感
- 绕行意愿
- 排队容忍
- 天气敏感
- 购买/服务需求倾向
- 校准状态

### 3. 行为程序层

每条 `behavior_program` 至少包含：

- 脚本名称
- 适用人群
- 触发条件
- 初始节点
- 目标节点
- 路径偏好
- 停留动作
- 消费动作
- 服务动作
- 放弃条件
- 外溢条件
- 概率参数
- 来源依据
- 待复核项

示例只是帮助理解，不应写死：

- 晨跑补给：早高峰、运动人群、5km后口渴、饮水/运动饮料需求、若排队过长则放弃。
- 亲子休息：周末、儿童疲劳、需要座位/遮阴/轻餐、若附近没有则外溢到商业街。
- 午间咖啡：工作日、周边白领、时间压力高、偏好快取、价格中等、超过绕行距离则放弃。

### 4. 概率计算层

不要让 DeepSeek 直接给最终概率。概率由 Python 计算，DeepSeek 只生成候选解释和缺口说明。

基础概率来源：

- 人群权重
- 时段权重
- 节假日/天气/活动权重
- 距离衰减
- 排队/拥挤惩罚
- POI供给承接能力
- 价格/预算匹配
- 运营时间匹配
- 证据置信度

输出不应只有一个分数，应包含：

- 推进优先级
- 为什么是这个优先级
- 关键依据
- 当前缺口
- 具体建议
- 待复核动作
- 不确定性来源

### 5. 状态-行为链验证层

检查单个虚拟人是否合理：

- 状态是否和行为一致。
- 路线是否可达。
- 需求触发是否有因果链。
- 消费/放弃是否符合预算、排队和距离。
- 服务设施需求是否和人群状态匹配。

注意：这里验证的是资料、状态、行为、空间和消费选择之间是否能自洽、能复跑、能被证据纠错；不是把“看起来像真人”当作方法完成标准。用户要求的“模仿人类”只放在后续网页可用性测试中。

### 6. 宏观验证层

检查总体仿真是否与真实数据一致：

- 时段客流曲线是否接近真实峰值。
- 热门业态需求是否和 TGI/消费偏好一致。
- POI供给是否和高德/现场清单一致。
- 运动、亲子、白领、老人、游客等人群比例是否有来源。
- 结果是否能解释已知供需缺口，而不是只制造新结论。

## 三、DeepSeek 的正确角色

DeepSeek 可以做：

- 生成候选人物状态。
- 生成行为程序草稿。
- 把用户上传资料转成待复核摘要。
- 解释仿真结果。
- 提出待补资料清单。
- 生成业务报告初稿。

DeepSeek 不可以做：

- 写入 checked 证据。
- 直接决定最终排名。
- 直接给收益预测。
- 在没有真实数据时补空。
- 把自己的推测写成已经验证。

## 四、界面改造方向

页面需要从“说明书”变成“业务工作台”：

- 人物仿真入口：新增“人群与行为程序”工作区。
- 支持新增/编辑/删除人群组。
- 支持新增/编辑/删除行为程序。
- 支持把上传资料标记为“采用/暂不采用/需补说明”。
- 支持运行场景：工作日、周末、活动日、极端天气、竞品变化。
- 支持查看结果：人群状态、行为路径、需求触发、供需缺口、建议动作。
- 支持折叠技术信息，默认给业务解释。

## 五、节点和分数的改造

用户指出“节点视觉有问题、打分意义不详”，判断成立。

改造原则：

- 分数不应做主视觉。
- 主视觉改为：`优先推进 / 先补证再议 / 暂缓推荐`。
- 每个节点要显示“为什么”和“下一步怎么做”。
- 详细解析放在折叠区，包含：资料依据、门禁缺口、POI关系、业务建议、待复核问题。
- 若仍保留分数，只作为辅助解释，不作为最终排名。

建议文案：

- “当前推进优先级”
- “待补资料与决策动作”
- “决策前置条件”
- “为什么现在还不能下结论”

## 六、验证计划

最少验证项：

- 文件存在和编码校验。
- DeepSeek API smoke。
- 高德 API smoke。
- PDF 表格验证。
- 人群状态 schema 校验。
- 行为程序 schema 校验。
- 概率参数边界校验。
- 供需缺口输出校验。
- 前端页面人工视觉检查。
- Selenium/Browser 连续使用检查。
- 报告生成检查。
- 真实资料回放检查。

真正进入“仿真可信”前，还需要：

- 用至少一个真实项目资料包回放。
- 对比真实客流峰值。
- 对比已知 POI 供给。
- 对比 TGI/消费偏好。
- 记录不能解释的偏差。
