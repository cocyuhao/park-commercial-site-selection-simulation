# 老板方法资料全盘吸收后的系统重基线（2026-06-04）

> 状态：主控判断稿。  
> 用途：约束后续代码、页面、报告、DeepSeek 任务和 GitHub 同步。  
> 关键口径：老板六份资料不是“缺口补充”，而是会改变仿真方向、工作量和旧文件可信度的上层方法材料。旧成果不能按旧完成度继续使用，必须先按本文重审。

## 0. 本次纠偏

用户指出得对：此前整理仍偏“看过资料后补几个缺口”，没有把六份资料和论文当成系统重构信号。正确理解应改为：

1. 这些资料改变的是系统主线，不只是补充参考文献。
2. 仿真工作量显著变大，因为必须从 POI/TGI 缺口公式扩展到人群状态、行为程序、空间移动、消费选择、运营约束和真实校准。
3. 旧文件中的“完成”“闭环”“仿真已跑通”只能代表旧标准下的产物，不能自动继承到新标准。
4. 所有论文都有价值，哪怕不直接采用，也至少会成为禁用边界、验证要求或风险提示。
5. DeepSeek 可以便宜地做候选和草稿，但系统设计、真实校准、最终判断和 checked 证据不能交给它。

本文取代“按缺口补材料”的工作方式。后续任何实现都必须回答：它落在本文哪一层，输入是什么，输出是什么，如何复核，失败时如何降级。

## 1. 六份老板资料的共同方向

六份资料不应被武断合成一个已完成系统，但方向是一致的：

- 不让 LLM 直接编故事。
- 不让 LLM 直接打最终分。
- 不让页面把 dry-run 写成完整仿真。
- 不把“看起来像人”当作“人类行为可验证”。
- 不把多 agent 数量当作多样性或真实性。
- 不把 POI/TGI 缺口公式当作最终商业决策。
- 必须把真实证据、人的潜在状态、行为脚本、空间路径、消费选择和宏观校准串起来。

### 1.1 GameLook / SSR / Agent Bank

吸收点：

- 合成用户只能做早期方向预警，不能替代真人、现场和真实业务数据。
- 不要直接问模型要 1-5 分或 1-100 分；应先生成自然语言动机、担忧、购买/放弃原因，再映射为优先级或选择概率。
- “1000 个智能体”重点不是数量，而是记忆素材、真实反馈和可验证对照。

落地约束：

- 节点主视觉不显示裸分数。
- 报告必须区分“已确认事实、早期信号、待复核判断、禁止判断”。
- 资料池要标明资料服务于哪类人群状态、行为程序、空间输入或校准目标。

### 1.2 ROTE / Modeling Others' Minds as Code

吸收点：

- 行为应被表达为可编辑、可复核的 behavioral programs，而不是每次让 LLM 临场猜。
- LLM 只生成候选程序；权重、约束、执行、复跑和验证应由本地系统完成。
- 没有真实轨迹时，Bayesian inference / Sequential Monte Carlo 只能作为后续方向，不能伪称已实现。

落地约束：

- 每个行为程序至少包括：适用人群、触发条件、初始状态、动作序列、路径偏好、消费动作、放弃条件、外溢条件、证据来源、校准状态。
- 用户必须能在页面上看见、修改、采用、放弃或锁定这些行为程序。
- DeepSeek 生成的行为程序必须过 schema 和本地校验，默认 `needs_review`。

### 1.3 HumanLM / 潜在状态对齐

吸收点：

- 模拟用户不是模仿回复表面，而是先建 latent state。
- 本项目的人群不能只有“亲子、白领、游客、老人”标签，必须有目的、疲劳、时间压力、预算、同行、绕行意愿、风险容忍等状态。
- LLM judge 可以做草评，但不能做最终门禁。

落地约束：

- AI 工作台和节点解释必须呈现 `state -> behavior -> demand -> advice`。
- 任何“人物仿真准确性”都先看状态是否充足，而不是看回答是否像人。
- 页面需要人群状态 CRUD，不能把状态写死在 prompt 或后端常量里。

### 1.4 社区高清社会仿真 / RL + LLM

吸收点：

- 高质量社会仿真依赖真实环境、真实人口/行为数据、动作空间、奖励设计和宏观校准。
- LLM 不是主引擎，而是微观合理性奖励或解释层。
- 宏观统计奖励和真实数据对齐是仿真可信的核心，不是 UI 截图或 Selenium 可替代的。

落地约束：

- 当前 `60_model/simulation/engine.py` 只能叫结构化 dry-run，不是完整仿真。
- 没有 DWG/GIS 几何、入口客流、分时客流、停留、转化率、收益成本和运营授权前，不得宣称 P4 完整仿真。
- 后续验证目标应包括客流曲线、节点停留、空间热力、需求分布、外溢、转化、收益成本等，而不是单次文本评分。

## 2. 21 篇外部论文/资料的系统用途

这些论文不是装饰性引用。每一类都至少给系统一个采用点或禁止点：

| 类别 | 代表条目 | 系统用途 |
|---|---|---|
| LLM 用户/人群模拟 | HumanLM、Out of One Many、LLMs to Simulate Multiple Humans、1000 people agents | 要求人群分布、状态和真实样本对照；禁止平均人和扁平画像 |
| 行为程序/生成式智能体 | ROTE、Generative Agents、LLM agent survey | 要求记忆、计划、行为程序和历史会话；禁止临场编故事 |
| LLM/ABM 风险批判 | causal inference critique、generative ABM review、identity flattening、diversity-of-thought | 要求因果谨慎、复现、校准和多样性；禁止把 plausibility 当 validity |
| 空间/路径仿真 | Social Force、RVO、MATSim | 要求路径、容量、拥挤、避障和活动链；禁止把静态地图或前端动画当空间仿真 |
| 商业/消费选择 | Huff、Gravity、Logit、Retail complementary goods ABM | 要求吸引力、距离衰减、选择概率、互补/替代关系；禁止单节点孤立打分 |

后续如果增加论文，必须按同样方式进入系统：不是“我读过”，而是转成对象、门禁、禁用边界或测试。

## 3. 新系统分层

### 3.1 证据层

输入：

- `evidence_ledger.csv`
- PDF 表格、TGI、客流、热门到访 POI
- 高德 POI、路径、边界和地图上下文
- 用户上传计划、DOCX、PDF、图片和现场反馈
- 老板方法资料和论文方法

输出：

- checked / needs_review / draft / conflict 四类证据状态
- 每个数字和判断的来源、方法、置信度和复核状态

禁止：

- DeepSeek 直接写 checked。
- PPT 或方法资料中的示例数字直接当真实商业证据。

### 3.2 人群潜在状态层

对象：

- `persona_group`
- `latent_state_profile`
- `state_transition`
- `memory_source`

必须包含：

- 到访目的、身体状态、时间压力、预算敏感、同行关系、绕行意愿、排队容忍、风险容忍、证据引用、缺失输入。

页面要求：

- 用户能新增、编辑、采用、放弃、删除、锁定。
- DeepSeek 只能生成候选状态，不得替用户采用。

### 3.3 行为程序层

对象：

- `behavior_program`
- `demand_trigger`
- `action_sequence`
- `abandon_condition`
- `spillover_condition`

必须包含：

- 触发条件、状态前提、动作序列、路径偏好、消费/服务动作、失败条件、外溢去向、校准状态。

页面要求：

- 不只显示“建议做饮品点”，而要显示为什么某类人会走到这里、什么时候会买、什么时候会放弃。

### 3.4 空间运动层

对象：

- 入口、节点、路网、障碍、距离、坡度/遮阴、容量、排队、可达性、地图边界。

当前状态：

- 高德地图和 POI 是空间线索。
- DWG/GIS 未闭合时不能做精确路径和面积推断。
- 静态底图只能是兜底展示，不能被当成仿真底层。

### 3.5 消费选择和供需层

对象：

- `choice_probability`
- `demand_score`
- `supply_score`
- `gap_index`
- `complement_substitute_relation`
- `operation_feasibility`

处理：

- POI/TGI 缺口公式作为需求/供给底座，不是最终推荐。
- 选择概率需要由状态、行为程序、空间成本、排队、价格、时段和竞品共同决定。
- 分数如果保留，默认折叠，并说明它只是当前资料下的讨论优先级。

### 3.6 运营和约束层

对象：

- 营业时间、授权边界、补货、人员、水电、施工、季节、活动、合规、成本、收益。

意义：

- 很多商业机会不是“需求高”就能做；若缺运营授权、成本和补货条件，只能是机会草案。

### 3.7 校准和验证层

微观门禁：

- 行为程序是否符合状态和场景。
- 路径、排队、距离、放弃是否合理。
- DeepSeek 草评只能给合理性解释，不能放行。

宏观门禁：

- 分时客流、入口客流、节点停留、热力图、业态需求、外溢、转化率、收益成本和真实资料对齐。

候选指标：

- SARIMA / 趋势一致性
- SSIM / 时空热力形态
- KL-Divergence / 分布差异
- DTW / 曲线形态
- 相关系数 / 峰谷时段
- 现场/运营复核

### 3.8 UI 与报告层

页面不是展示旧结果的壳，而是让业务人员参与复核：

- 项目总览：显示哪些输入闭合、哪些仍阻塞。
- 资料池：显示资料被用在哪个模型对象上。
- 节点页：显示优先级、具体建议、理由、缺口和复核动作。
- AI 工作台：围绕项目综合、节点、资料和报告工作，而不是测试面板。
- 报告：商业工作报告，不是 AI 日志，不得把草案写成结论。

## 4. 旧文件可信度重新判定

| 类型 | 新状态 | 处理 |
|---|---|---|
| 证据台账、PDF 表格、抽取脚本、API 健康脚本 | 仍可信底座 | 保留，但只能证明资料处理和工具可用 |
| P6 页面、AI 工作台、资料池、报告按钮、Selenium 截图 | 产品壳可用 | 不证明模型真实，只证明交互可用 |
| P2 persona、behavior、validation CSV | 草稿候选 | 需要映射到 schema 并人工/脚本复核 |
| 旧 DeepSeek 输出 | 历史草稿 | 35 个旧输出没有符合新 envelope，必须 adapter 或降级 |
| P4 完整仿真、最终排序、ROI、最终推荐 | 不可信 | 统一降级为旧标准 dry-run 或草案 |
| 节点裸分数 | 不适合作主视觉 | 改成推进优先级、建议、理由、缺口和复核动作 |

## 5. DeepSeek 的新位置

DeepSeek 可做：

- 批量资料摘要
- 人群状态候选
- 行为程序草稿
- 节点解释草稿
- 微观合理性草评
- 报告语言草稿

DeepSeek 不得做：

- checked 证据
- 最终排名
- ROI/收益预测
- 最终推荐
- 仿真完成声明
- 用户不可编辑的隐藏决策

每个 DeepSeek 任务都要有：

- 输入文件列表
- JSON schema
- 字段白名单
- `draft/needs_review` 状态
- 本地校验脚本
- 失败降级策略
- 用户可见的采用/放弃路径

1. 人群状态库 CRUD。
2. 行为程序库 CRUD。

4. 旧 DeepSeek 输出 envelope adapter。
5. 节点解释从裸分数改为 `state -> behavior -> demand -> advice`。
6. 供需缺口公式降级为一个层，不再做最终结论。
7. 空间运动门禁：DWG/GIS/入口/路径/容量缺口显式化。
8. 宏观验证计划和指标占位。
9. 报告模板重写：已确认、早期信号、待复核、禁止判断分开。

### 6.2 可以保留但要改口径

1. AI 工作台：保留产品壳，接入人群/行为/报告对象。
2. 项目总览：保留状态卡，但状态要来自证据和门禁。
3. 资料池：保留上传/解析/采用动作，但补齐模型对象引用。
4. 同事 POI/TGI：保留缺口计算思路，接到消费选择层，不上升到完整仿真。
5. Selenium：继续做产品可用性验证，但不再当模型准确性验证。

### 6.3 暂不做

1. Unreal Engine / 高清数字孪生复刻。
2. PPO/GRPO 训练。
3. Sequential Monte Carlo 权重更新。
4. 实时逐游客调用 DeepSeek。
5. 未闭合数据下的 ROI、收益、最终排名。

## 8. 一句话交接

现在这个项目不是“POI/TGI 缺口 + 好看的 AI 工作台”。它已经变成“证据驱动的人群状态与行为程序仿真系统”。旧结果不是没用，但要全部降级重审；DeepSeek 不是不用，但只能做便宜的候选生成和解释；页面不是摆结果，而是让业务人员看懂、采用、放弃、复核和推进。
