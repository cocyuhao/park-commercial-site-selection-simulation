# 全局 AI 仿真决策系统设计重基线（2026-06-04）

> 本文件不是 P6 页面美化指南，而是整个“AI 驱动仿真决策系统”的产品、交互、视觉和方法落地重基线。
> 后续修改 `90_p6_expert_dashboard/`、仿真对象、资料池、节点推进、AI 工作台和报告生成时，都应先按本文判断。
> 公园商业选址是当前已接入的业务场景，不是系统能力边界；后续命名、对象模型和交接文件不得把全局系统写死为单一公园商业工作台。

## 1. 当前判断

原界面需要大改，但不是无保护推翻。

原因不是“页面不好看”这么简单，而是旧界面背后的功能模型已经和当前目标冲突：

- 旧界面以“节点、分数、门禁、草案展示”为中心，容易让用户误以为已有结论。
- 新目标是“AI 驱动仿真决策系统”：资料、方法对象、人群状态、空间约束、仿真输入、证据缺口、AI 解释、报告输出必须串成一条链。
- 老板资料和 2026 外部资料都指向更强的人机协作与 agentic workflow，不适合继续把 AI 做成一个孤立问答框。
- 用户要求“自由加减、不要写死”，意味着系统必须支持可编辑对象、可采用/放弃/锁定、可追溯依据，而不是固定模板。
- 旧视觉里的系统默认按钮、技术词、裸分数、密集说明、所有内容平铺，会破坏业务人员对系统专业性的信任。

因此后续采用“保留能力、重组主链”的策略：保留已跑通的上传、地图、节点、AI、报告、仿真检查接口；逐步把用户可见逻辑重构为“目标 -> 对象 -> 依据 -> 动作 -> 复核 -> 报告”。

对本项目的可采用部分：

1. 先定义产品调性，不要急着画页面。
   - Flowus 的重点不是复制某个风格，而是先明确目标用户、核心感受、视觉隐喻和反目标。
   - 本项目的全局调性应是“可信、克制、专业、可审计、能推进”，不是炫技、不是儿童化、不是营销落地页。

2. PRD / 信息架构先行。
   - 不应继续先堆页面组件，再补逻辑。
   - 后续要先明确全局对象：项目、资料、方法对象、节点、人群状态、POI、仿真任务、AI 会话、报告。

3. 参考图和情绪板只用于抽取原则。
   - 可以借鉴高级感、真实材质、空间留白、动效节奏。
   - 不能为了视觉高级而加入与公园商业仿真无关的背景视频、装饰图、强营销 Hero。

4. 动效用于反馈，不用于装饰。
   - 对象采用/放弃/锁定、资料入池、AI 生成报告、仿真任务进度，可以有轻量状态动效。
   - 地图加载、资料解析、AI 回复必须有稳定状态，不允许闪烁、清空或让用户误以为操作丢失。

5. 去 AI 味不是口吻问题，而是“真实产品感”。
   - 文案要像业务工作台：说明范围、状态、依据、下一步。
   - 视觉要像成熟工具：稳定布局、明确状态、可操作对象、少解释、多动作。

不采用或谨慎采用的部分：

- Flowus 示例偏零代码、移动端、个人 App；不能照搬成“小白教程式界面”。
- 背景图/视频的做法更适合营销页或情绪型产品；本项目是决策系统，第一屏应是工作面。
- Pencil/Cursor 的流程可作为设计-实现协同参考，但不替代本地验证、Selenium、证据链和代码门禁。

## 4. 全局产品对象模型

后续系统不再以页面为中心，而以对象为中心：

| 对象 | 用户能做什么 | 系统必须显示什么 |
|---|---|---|
| 项目 | 确认项目范围、位置、阶段 | 当前目标、资料闭合度、下一步推进动作 |
| 资料 | 上传、采用、放弃、删除、解释用途 | 文件名、类型、状态、是否采用、解析说明、关联对象 |
| 方法对象 | 采用、放弃、锁定、编辑 | 方法来源、适用范围、缺口、具体建议 |
| 节点 | 新增、编辑、比较、进入 AI 对话 | 推进优先级、依据、缺口、具体动作，不显示裸分数 |
| 人群/行为 | 选择 persona、调整触发、进入仿真 | 状态假设、约束、来源、是否已校准 |
| 地图/POI | 搜索、拖拽、筛选、关联节点 | 位置、边界、POI 语境、加载状态、错误不清空 |
| 仿真任务 | 运行、查看进度、复核结果 | 输入对象、校准目标、结果状态、失败原因 |
| AI 会话 | 新对话、项目综合、单节点、生成报告 | 当前范围、历史、可追溯引用、用户可不同意 |
| 报告 | 生成、查看、下载、复核 | 摘要、依据、缺口、待复核判断、推进事项、附录 |

## 5. 全局界面原则

1. 工作台第一屏不是宣传页。
   - 第一屏必须回答：项目在哪一步、有什么资料、哪些对象可用、下一步做什么。

2. 可见状态必须对应真实数据。
   - 状态卡不能写死；完成项目、上传资料、采用对象、生成报告后必须变化。

3. 不再暴露后端语言。
   - 用户界面不得出现 raw、payload、debug、traceback、external_preview_only、needs_review 等内部词。

4. 不再用裸分数讲结论。
   - 分数若保留，只作为内部排序辅助；用户看推进优先级、依据和建议。

5. 默认折叠复杂内容。
   - 背景、依据、缺失资料、原始对话、技术验证只在用户展开时显示。

6. 每个 AI 动作必须有范围和后果。
   - 项目综合、节点分析、资料解析、生成报告、运行仿真都必须告诉用户当前输入范围。

7. 用户应能不同意 AI。
   - 每个重要对象都应支持采用、放弃、编辑、锁定或删除。

8. 动效服务于反馈。
   - 只在状态切换、加载、完成、错误、展开/收起时用轻量动效。

## 6. 界面大改路线

### 阶段 A：保能力，换主链

- 已开始：新增“人物仿真对象池”，把 `choice_probability` 和 `simulation_validation_target` 落成可采用/放弃/锁定/删除对象。
- 已开始：节点从“打分”转为“推进优先级 + 依据 + 建议”。
- 下一步：项目总览改成“全局推进台”，显示项目、资料、方法对象、节点、地图、仿真、报告的链路状态。

### 阶段 B：重构 AI 工作台

- 左侧：项目 / 历史 / 新对话 / 范围切换。
- 中间：阅读区，默认项目综合。
- 底部：稳定输入框，工具包括添加资料、项目综合、节点分析、报告润色、生成报告。
- 输出：按“理解状态、可判断、不能判断、需补资料、推进事项”组织。

### 阶段 C：重构资料池和节点池

- 资料池不再是文件列表，而是可采用对象池。
- 节点池不再是分数列表，而是推进对象池。
- 节点卡只显示短状态；详情页显示依据和建议。

### 阶段 D：仿真与报告闭环

- 仿真任务必须显示采用了哪些对象、哪些目标尚未校准。
- 报告生成必须知道是项目综合报告还是单节点报告。
- 报告不写机器字段，保留附录引用。

## 7. 立即约束后续代码

- 新增用户可见 UI 前，先问：它属于哪个全局对象？用户能对它做什么？
- 新增 AI 输出前，先问：它是否能被采用/放弃/锁定/追溯？
- 新增状态卡前，先问：它是否随数据变化？
- 新增分数或排序前，先问：能否改为优先级、依据、建议？
- 新增文案前，先扫：是否像说明书、是否像后端日志、是否把 agent 思考强加给用户？
- 新增设计细节前，先确认：它解决了反馈、理解、信任还是推进问题？
