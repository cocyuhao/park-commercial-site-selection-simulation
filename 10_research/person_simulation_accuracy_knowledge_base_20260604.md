# 人物仿真准确性知识库与工程路线（2026-06-04）

> 状态：可交接知识库 v1。  
> 主线：以用户自己的理解为主，同事的 POI/TGI 方向作为供需缺口辅助层。  
> 目标：提高人物仿真准确性，并把 DeepSeek 放在受限位置，让用户能在网页中增删改查和掌控假设。

## 0. 本轮结论

人物仿真不能理解成“让 DeepSeek 捏几个人，然后问他们怎么走、怎么买”。那会很快、很像样，但不可靠。正确路线是：

1. 先用真实资料、地图、时间、天气、客流、POI、TGI、活动场景和用户补充资料构建可编辑的“人群状态”。
2. 再把人群状态转成可审查的“行为程序”：什么时候来、为什么停、为什么绕、为什么买、为什么放弃。
3. 再进入空间路线和消费选择：路线不是一句话，必须受入口、节点、距离、拥挤、排队、服务设施、营业时间和同行人限制。
4. 再做供需缺口和收益/时段策略：先有“谁在什么时间为什么需要什么”，再判断是否放饮水机、售卖机、咖啡、轻食、亲子零售或服务设施。
5. DeepSeek 只负责草稿和解释，不负责最终判断。用户必须能改、删、锁定、弃用每一类假设。

一句话：人物仿真准确性的核心不是“AI 更会说”，而是“状态、行为、选择、空间、供需、校准、用户控制”都被拆开并可复核。

## 1. 同事方案的吸收边界

同事写的方向：

- 人物行为逻辑虚拟仿真。
- 供需缺口，参考 `https://github.com/Hiromitsu1207/POI_TGI_Calculator`。

选择性吸收：

| 可吸收部分 | 为什么有用 | 本项目处理 |
|---|---|---|
| TGI -> demand profile | 能把报告偏好转成业态需求权重 | 保留为“需求层”，不当人物完整行为 |
| POI -> supply profile | 能形成现有供给画像 | 保留为“供给层”，并记录 POI 来源和距离 |
| `gap_index = (demand - supply) / (demand + supply)` | 比裸差值更适合比较不同业态 | 可作为缺口指标，但不等于最终推荐 |
| observed / inferred supply 分层 | 真实 POI 与推断服务设施分开，避免互相抵消 | 应吸收进资料池和报告 |
| operation_suggestions / spatial_suggestions | 有助于把缺口转成动作 | 必须接入人物状态和时段，不可直接模板输出 |
| PDF/TGI 解析诊断 | 能标记低置信度资料 | 可并入资料池状态 |

不能照搬：

- 不能把单个 tourist profile 的 TGI 当作真实人群分布。
- 不能用 OpenAI/LLM 直接生成最终 TGI 或运营建议。
- 不能把“供需缺口”当作人物仿真本身。
- 不能只用年龄、性别、收入生成画像。
- 不能把 P0/P1/P2 priority 展示成最终选址排名。

## 2. 文献和成熟产品带来的约束

| 来源 | 启发 | 本项目落点 |
|---|---|---|
| MATSim | 大规模 agent 不是逐人问 LLM，而是活动计划、出行网络、迭代优化 | 做“日程/活动链/路线”结构，不让 DeepSeek 每步自由发挥 |
| SUMO | 微观、多模式、连续交通仿真强调网络、路线和需求数据 | 路线仿真要依赖真实地图网络和时间窗口 |
| ActivitySim / activity-based model | 人的出行来自活动需求，不是随机移动 | 先定义目的：跑步、遛娃、通勤绕行、约会、观景、补给 |
| UrbanSim | 微观仿真必须透明、行为导向、能解释政策/设施干预 | 页面要展示假设、状态、缺口和干预影响 |
| PTV Viswalk | 成熟行人仿真关注宽度、等待区、密度、瓶颈、排队、延误 | 公园节点要有容量、拥挤、等待、通过时间 |
| AnyLogic Pedestrian Library | 可用来分析商场/公共空间的人流、商业位置、紧急疏散 | 我们应把商业点位价值与 footfall/停留/转化挂钩 |
| Generative Agents | LLM agent 需要记忆、反思、计划，不能只靠 persona 标签 | DeepSeek 只能生成草稿记忆/计划，不可最终化 |
| LLM-ABM surveys | LLM agent 容易黑箱和不可校准 | 必须本地 schema + 真实数据校准 |
| Causal critique of LLM simulated subjects | LLM 模拟人会混入未控制变量 | 必须显式列出假设和缺失变量 |
| Time geography | 人的行为受能力、耦合、权威约束限制 | 加入体力、同行、开放时间、管理权限等约束 |
| Huff / gravity / logit | 消费地点选择可用距离、吸引力和替代方案概率化 | 不用“喜欢就买”，用选择概率 |

## 3. 人物仿真准确性的 12 个层级

### 3.1 资料层

输入：

- 老板六份方法资料。
- PDF/DOCX/PPT/长图。
- 高德 POI、路线、边界、静态/JS 地图。
- 用户上传计划、节点、资料说明。
- 可选实时数据：天气、节假日、活动、客流、交易、设备库存、运维记录。

准确性要求：

- 每个数据源都有状态：已采用、待解析、解析失败、暂未采用、需要补充说明。
- 每个结论都有来源。
- 没有来源时不能伪装成事实。

### 3.2 人群状态层

不再只写“年轻白领/亲子/老人”。应包含：

- 目的：跑步、散步、亲子、通勤、约会、社交、观景、补给。
- 时间压力：是否赶时间。
- 预算敏感：愿不愿意多花钱。
- 疲劳：走了多久、是否需要休息。
- 同行：一个人、情侣、家庭、朋友、团体。
- 绕行容忍：愿不愿意多走。
- 排队容忍：能不能等。
- 服务敏感：厕所、饮水、母婴、停车、遮阴。
- 环境偏好：安静、热闹、避晒、临水、开阔。

### 3.3 行为程序层

行为不是一句“会买咖啡”。要写成程序：

- 触发条件：早晨跑步后、夏季高温、亲子停留超过 60 分钟、排队超过 8 分钟。
- 可选动作：通过、靠近、停留、购买、比较、绕行、离开、休息。
- 失败条件：排队长、价格高、距离远、同行反对、夜间关闭、补货不足。
- 输出信号：停留时间、购买概率、路线偏移、节点拥挤、服务缺口。

### 3.4 时间层

至少区分：

- 清晨、上午、午间、下午、傍晚、夜间。
- 工作日、周末、节假日。
- 运动高峰、亲子高峰、旅游高峰、通勤绕行高峰。
- 营业、关闭、补货、维护、临时活动。

### 3.5 空间层

至少区分：

- 入口、出口、路径、湖边、广场、活动区、服务节点、停车/换乘点。
- 距离、坡度、遮阴、视线、临水、拥挤、座椅、照明。
- 是否能摆摊、是否能接水电、是否有消防/城管/物业限制。

### 3.6 需求触发层

需求不只是“想喝水”。应有：

- 生理：渴、饿、累、上厕所、带娃、避雨、避暑。
- 社交：等待、合影、约会、朋友聚集。
- 活动：跑步、骑行、儿童游玩、文化体验。
- 信息：找路、问询、查看活动。
- 消费：补给、轻食、咖啡、茶饮、文创、亲子。

### 3.7 供给层

来自：

- 真实 POI。
- 公园内已有设施。
- 用户手动新增设施。
- 规则推断设施。
- 未来候选设施。

必须区分真实供给和推断供给，不能让推断供给抵消真实缺口。

### 3.8 选择概率层

建议采用三类模型：

- Huff / gravity：距离和吸引力决定到达概率。
- Logit / discrete choice：价格、等待、距离、品牌、同行约束共同影响选择。
- Rule gate：管理权限、营业时间、容量、天气直接阻断不可行方案。

### 3.9 运营动作层

输出不应只是“这里缺咖啡”。要给动作：

- 设施类型。
- 节点位置。
- 营业/关闭时间。
- 补货时间。
- 产品结构。
- 人员/维护要求。
- 风险。
- 待复核数据。

### 3.10 收益层

收益必须后置。先有：

- 人群量。
- 到达概率。
- 停留概率。
- 购买概率。
- 客单价。
- 复购/季节性。
- 成本、租金、分成、补货、人力、设备折旧。

缺任一关键项，就只能输出“收益缺口待复核”，不能输出 ROI。

### 3.11 校准层

用真实数据校准：

- 客流计数。
- 手机信令/热力/高德数据。
- 交易流水。
- 设备销量。
- 现场观察。
- 问卷/访谈。
- 活动日记录。
- 天气与节假日。

校准指标：

- 时段流量误差。
- 节点流量误差。
- 停留时间分布。
- 路径分布。
- 购买率误差。
- 缺口排序稳定性。

### 3.12 用户控制层

网页必须给用户自主权：

- 新增/删除/编辑人群画像。
- 新增/删除/编辑行为程序。
- 新增/删除/编辑节点、设施、营业时间、供给能力。
- 选择资料是否采用。
- 锁定某条假设，不让 DeepSeek 改。
- 放弃某条 AI 草稿。
- 标记“已复核/待复核/不采用”。
- 查看每条建议的依据和缺口。

## 4. DeepSeek 在人物仿真中的正确位置

DeepSeek 可以：

- 把用户自然语言转成画像状态草稿。
- 把资料摘要转成待复核指标。
- 生成行为程序候选。
- 解释某个节点为什么可能适合或不适合。
- 把报告语言写得更像业务报告。

DeepSeek 不可以：

- 决定最终人群分布。
- 直接输出最终路线、最终消费、最终收益。
- 把缺失数据自动补齐。
- 把“合理故事”冒充真实仿真。
- 替用户决定是否采用资料。

## 5. 网页需要支持的 CRUD 对象

| 对象 | 用户能做什么 | DeepSeek 能做什么 |
|---|---|---|
| 资料 | 上传、采用、放弃、删除、补说明 | 摘要和字段建议 |
| 人群画像 | 新增、编辑、删除、锁定权重 | 生成草稿 |
| 行为程序 | 新增、编辑、禁用、排序 | 提供候选动作和失败条件 |
| 时间场景 | 新增时段、设置开放/关闭/补货 | 给出草案 |
| 空间节点 | 新增、删除、移动、设置属性 | 解释节点潜力 |
| 供给设施 | 新增、删除、设置容量/营业时间 | 建议设施类型 |
| 校准目标 | 录入真实数据、设权重 | 提醒缺口 |
| 运营方案 | 采用、放弃、比较 | 生成待复核方案 |

## 6. 提高准确性的方案清单

1. 用状态画像替代浅标签画像。
2. 用行为程序替代一次性问答。
3. 用用户可编辑 schema 替代写死页面字段。
4. 用时段和天气拆分需求。
5. 用入口、路径、节点和出口拆分空间。
6. 用真实 POI 和推断设施分层。
7. 用距离、排队、价格、品牌、同行约束建立选择概率。
8. 用资料状态和证据链防止模型幻觉。
9. 用 DeepSeek 生成候选，但所有输出进入 `draft/needs_review`。
10. 用真实客流、交易、设备销量和现场观察校准。
11. 用用户锁定/弃用机制保留人类控制权。
12. 用报告把“已确认、待复核、缺口、下一步”分开，而不是包装成确定结论。
